Overview

Add team-invite capability so workspace owners can send signup links to teammates, assign their role on creation, and revoke pending invites before they're accepted. Tokens expire after seven days and are single-use. The flow touches the database (new invites table), two API endpoints, and three new UI screens (invite form, pending list, accept landing).

Acceptance Criteria

0/0
  • Workspace owner can submit an email + role and receive a confirmation toast.
  • Pending invites are persisted in invites with a 7-day TTL.
  • Accepting a valid token creates the user with the assigned role and consumes the invite.
  • Expired or already-consumed tokens return 410 Gone with a friendly error page.
  • Owners can revoke a pending invite from the list, which immediately invalidates the token.
  • Needs clarificationRate limiting on the invite endpoint — per-workspace cap vs. per-user cap is undecided.

Decision Log

DateDecisionRationale
2026-05-08 7-day token TTL Long enough that invites don't expire while teammates are on PTO; short enough that stale invites don't accumulate.
2026-05-09 Store token_hash, not the raw token If the DB leaks, tokens cannot be replayed. Email sends the raw token; only the hash is persisted.
2026-05-10 BullMQ over inline sending Mail delivery shouldn't block the API response. Project already runs Redis + BullMQ for other jobs.
2026-05-11 Empty state in v1 List exists from day one of the team feature; users will hit it with zero invites. Cheap to design upfront.
